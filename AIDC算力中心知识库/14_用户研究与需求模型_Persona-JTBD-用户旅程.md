# AIDC用户研究与需求模型：Persona、JTBD与用户旅程

版本：V1.0  
状态：研究框架，需用真实客户访谈持续验证

---

## 1. 研究目标

用户研究需要回答：

1. 谁购买、谁决策、谁使用、谁受益。
2. 用户在什么场景下遇到什么问题。
3. 当前如何处理，为什么现有方式不够好。
4. 哪些问题最值得优先解决。
5. 客户愿意为什么结果付费。

AIDC是典型ToB多角色产品，不能只写“目标用户是运维人员”。同一功能可能由平台管理员使用、由算力中心负责人决策、由财务验证价值、由安全团队否决。

# 2. 购买委员会

| 角色 | 目标 | 核心问题 | 决策权 |
|---|---|---|---|
| CIO/CTO | 提升AI基础设施投资回报 | 项目是否值得投入，是否可持续 | 最终决策 |
| 算力中心负责人 | 保证交付、利用率和SLA | 资源是否真正可售、可用、可运营 | 业务决策 |
| 平台/运维负责人 | 稳定运行和可维护 | 是否增加运维复杂度，能否快速排障 | 技术决策 |
| 财务/运营 | 收入、成本和对账 | 用量、价格、账单是否准确 | 价值验证 |

# 3. Persona一：算力中心负责人

## 基本信息

- 所属组织：IDC、运营商、城市智算中心或企业基础设施部门。
- 工作目标：保证资源交付、客户SLA、GPU利用率和经营结果。
- 使用频率：每日查看经营与风险总览，周/月度复盘。
- 专业程度：理解基础设施和业务，但不一定深入每个技术组件。

## 核心任务

- 判断当前可售GPU和未来容量。
- 跟踪客户交付、故障和SLA。
- 查看利用率、收入、成本和毛利。
- 决定扩容、采购和运维投入。

## 当前工具

- Excel和周报。
- DCIM或动环系统。
- Kubernetes/Slurm报表。
- 财务和合同系统。
- 人工汇总的经营驾驶舱。

## 主要痛点

- 不同系统指标口径不一致。
- GPU总量不等于真正可售资源。
- 故障影响客户和收入无法快速判断。
- 扩容依赖经验，缺少容量预测。
- 需要依赖多个负责人汇总数据。

## 关注指标

- 可售GPU、已售GPU和利用率。
- 任务成功率与排队时长。
- SLA达成率。
- 收入、成本、毛利和续费率。
- 重大故障和容量风险。

## 成功标准

在一个驾驶舱内看到资源、客户、故障和经营状态，并能够追溯指标来源。

# 4. Persona二：Kubernetes/平台管理员

## 基本信息

- 所属组织：平台工程、基础架构或SRE团队。
- 工作目标：保证集群稳定、资源可调度和版本安全。
- 使用频率：每日高频使用，故障时集中使用。
- 专业程度：熟悉Kubernetes，但可能只对部分网络、存储或GPU组件深入。

## 核心任务

- 接入和维护多个集群。
- 管理节点、插件、队列和配额。
- 排查Pod、网络、存储和控制面问题。
- 执行节点维护、升级和变更。
- 支持研发和AI团队解决基础设施问题。

## 当前工具

- kubectl、K9s。
- Prometheus、Grafana。
- ELK/Loki。
- Rancher、OpenShift或其他容器平台。
- GitLab、Argo CD、Jenkins。
- 工单和聊天工具。

## 主要痛点

- 告警多但根因不清楚。
- 日志、指标、事件和变更分散。
- 多集群版本和插件不一致。
- 高风险操作依赖少数专家。
- GPU、网络和存储问题跨团队。

## 关注指标

- 集群可用性。
- MTTD、MTTR。
- 告警降噪率。
- 节点和工作负载健康。
- 版本、配置和安全风险。

## 成功标准

十分钟内获取完整故障上下文，并安全执行经过验证的处置流程。

# 5. Persona三：SRE值班工程师

## 核心任务

- 接收并确认告警。
- 判断是否为真实故障。
- 确认影响范围和负责人。
- 执行临时恢复。
- 记录故障过程并推动复盘。

## 主要痛点

- 夜间收到大量重复告警。
- 无法快速区分根因和次生告警。
- 缺少业务、客户和SLA信息。
- Runbook散落，执行步骤不一致。
- 修复后难以验证是否真正恢复。

## 成功标准

平台将数百条原始告警聚合为少量故障事件，提供证据、影响面和安全处置建议。

---

# 6. Persona四：AI工程师/算法工程师

## 核心任务

- 创建Notebook和开发环境。
- 提交训练任务。
- 选择GPU规格、镜像和数据集。
- 查看排队、运行、日志和指标。
- 发布和观察推理服务。

## 主要痛点

- 不知道任务为何长期排队。
- 失败后难判断是代码还是基础设施问题。
- 环境、驱动和依赖不一致。
- GPU资源申请和审批周期长。
- 任务成本和资源效率不透明。

## 关注指标

- 任务排队时长。
- 训练成功率和运行时长。
- GPU利用率和显存使用。
- 推理延迟、吞吐和错误率。
- 项目配额和费用。

## 成功标准

能够自助申请资源、提交任务、定位常见问题，并获得明确的排队和成本解释。

---

# 7. Persona五：租户管理员

## 核心任务

- 管理团队成员和权限。
- 分配项目配额。
- 审批资源申请。
- 查看团队资源和费用。
- 处理续费和资源回收。

## 主要痛点

- 无法了解成员实际使用情况。
- 配额长期被占用但未产生价值。
- 预算和资源使用缺少关联。
- 到期资源、闲置资源未及时回收。

## 成功标准

能够按照团队和项目分配资源，控制预算，并及时识别闲置和异常使用。

---

# 8. Persona六：运营与财务人员

## 核心任务

- 管理产品、规格和价格。
- 审核订单和合同。
- 生成账单和对账单。
- 处理折扣、欠费、续费和开票。
- 分析收入、成本和毛利。

## 主要痛点

- 技术用量无法直接映射商业产品。
- 卡时、存储、流量和Token口径不一致。
- 对账依赖人工导表。
- 无法追踪账单调整原因。
- 产品毛利难以准确计算。

## 成功标准

用量可追溯、账单可解释、调整有审计，经营报表能够按客户和产品下钻。

---

# 9. Persona七：安全与合规人员

## 核心任务

- 审查租户隔离、权限和审计。
- 检查镜像、工作负载和集群基线。
- 跟踪漏洞和整改。
- 输出合规报告。

## 主要痛点

- RBAC关系复杂，难判断真实权限。
- 多集群安全标准不一致。
- 高风险配置无法及时发现。
- 研发和运维认为安全检查影响效率。

## 成功标准

风险可量化、整改有责任人和期限，高风险变更能够在执行前拦截。

# 10. 核心JTBD

JTBD格式：当某种情况发生时，我希望完成某项任务，从而获得某种结果。

| 编号 | JTBD |
|---|---|
| J01 | 当生产集群出现大量Pod重启时，我希望快速识别共同根因，从而避免逐个Pod排查和故障扩散。 |
| J02 | 当客户申请GPU资源时，我希望自动判断库存、配额和交付条件，从而给出可靠交付时间。 |
| J03 | 当训练任务长期排队时，我希望看到无法调度的具体原因，从而调整规格、配额或队列。 |
| J04 | 当GPU发生ECC或掉卡时，我希望立即知道受影响任务和客户，从而快速隔离资源并控制SLA影响。 |
| J05 | 当月末出账时，我希望技术用量自动映射到合同价格，从而减少人工对账和账单争议。 |
| J06 | 当集群准备升级时，我希望自动检查插件和API兼容性，从而降低升级失败风险。 |
| J07 | 当资源利用率偏低时，我希望区分真实闲置、资源碎片和配置浪费，从而采取正确优化措施。 |
| J08 | 当管理层询问智算中心经营状况时，我希望统一展示资源、收入、成本和SLA，从而支持扩容和经营决策。 |

# 11. 用户旅程一：处理生产故障

| 阶段 | 用户行为 | 当前痛点 | 产品机会 | 关键指标 |
|---|---|---|---|---|
| 发现 | 收到告警或客户投诉 | 重复告警多、业务影响不清 | 告警聚合、业务影响识别 | MTTD |
| 定位 | 查看Pod、节点、日志和指标 | 工具分散、上下文缺失 | 故障时间线、拓扑和证据链 | 定位时长 |
| 判断 | 联系研发、网络、存储团队 | 责任边界不清 | 根因置信度、专家规则 | 转派次数 |
| 处置 | 重启、隔离、回滚或扩容 | 操作风险高、步骤不一致 | Runbook、审批、影响预览 | 处置成功率 |
| 验证 | 查看指标是否恢复 | 容易只恢复表象 | 自动健康检查和SLA验证 | 恢复验证时长 |
| 复盘 | 整理时间线和原因 | 人工整理耗时 | 自动事故报告和知识沉淀 | 复盘完成率 |

# 12. 用户旅程二：申请并交付GPU资源

| 阶段 | 客户/运营行为 | 当前痛点 | 产品机会 |
|---|---|---|---|
| 选品 | 询问销售可用规格和价格 | 库存和价格不透明 | 产品目录、可售资源 |
| 申请 | 填写表格或发送邮件 | 信息不标准 | 标准规格和申请单 |
| 审批 | 多部门确认预算、配额 | 状态不可见 | 审批流和SLA |
| 匹配 | 运维人工寻找服务器 | 物理空闲不等于可交付 | 规格匹配和交付预检 |
| 开通 | 装机、驱动、网络、账号 | 人工步骤多 | 自动化交付工作流 |
| 验收 | 客户测试资源 | 验收标准不统一 | 交付检查和压测报告 |
| 使用 | 查看任务和用量 | 状态和费用不透明 | 资源实例和用量中心 |
| 续费/回收 | 人工提醒和处理 | 资源回收不及时 | 到期策略、续费和回收 |

# 13. 用户旅程三：月度计量与出账

```text
采集K8s/Slurm/网关/存储用量
→ 清洗、补偿、去重
→ 关联资源实例、租户、订单和合同
→ 应用价格、折扣和阶梯规则
→ 生成预账单
→ 运营审核异常
→ 客户对账
→ 正式账单和开票
→ 收款、欠费和续费
```

关键风险：

- 数据源中断。
- 任务与客户映射缺失。
- 重复采集。
- 时间区间和计费粒度不一致。
- 价格版本变更。
- 手工调整缺少审计。

# 14. 访谈提纲

## 决策者

1. 为什么现在建设或升级AIDC平台。
2. 最核心的经营和管理目标是什么。
3. 当前最担心的风险是什么。
4. 项目预算来自哪个部门。
5. 如何判断项目成功。
6. 哪些系统不能替换。

## 平台管理员/SRE

1. 当前管理多少集群、节点和GPU。
2. 最近三次重大故障是什么。
3. 排查过程中切换哪些工具。
4. 哪些操作最依赖专家。
5. 哪些告警最无效。
6. 当前升级和变更流程如何执行。

## AI工程师

1. 资源申请需要多长时间。
2. 任务为什么排队或失败。
3. 当前如何查看GPU使用和成本。
4. 最希望自助完成哪些操作。
5. 哪些限制影响训练或推理效率。

## 运营财务

1. 当前计量和账单数据来自哪里。
2. 每月有多少人工修正。
3. 最常见对账争议是什么。
4. 产品、合同和资源如何关联。
5. 如何计算成本和毛利。

# 15. 研究证据等级

| 等级 | 证据 | 使用方式 |
|---|---|---|
| E0 | 产品团队判断 | 只能作为假设 |
| E1 | 单个客户原话 | 作为线索，不能代表市场 |
| E2 | 多客户重复反馈 | 可进入需求池 |
| E3 | 工单、日志或运营数据 | 可支持优先级判断 |
| E4 | 客户愿意POC或付费 | 强需求证据 |
| E5 | 上线后指标显著改善 | 产品价值已验证 |

每条P0需求至少应达到E2；重大投资需求应尽量达到E3或E4。
