## 4. 核心总纲

AIDC项目不是单纯建设机房，也不是简单购买GPU服务器。它本质上是把电力、制冷、机房、服务器、GPU、高速网络、存储、Kubernetes/Slurm调度系统、AI开发平台、模型服务、客户门户、订单计费和运营分析连接为一条完整的“算力生产与经营闭环”。

AIDC运营管理平台的使命，是把复杂的基础设施和运维过程，转化为可监控、可调度、可运营、可售卖的平台能力。

六个核心价值：

- 看得见：设施、资产、算力、任务、能耗全量可视化。
- 管得住：告警、权限、工单、变更闭环管控。
- 调得动：GPU、服务器、存储、网络统一灵活调度。
- 用得好：降低排队、碎片与闲置，提高有效产出。
- 算得清：用量、成本、电费、账单、收入清晰可溯。
- 卖得出去：资源封装为产品，客户自助申请并自动计量计费。

面向MaaS和AI基础设施岗位，还需要增加两条贯穿产品和技术的主线。

### 技术约束到产品价值

```text
模型与硬件特性
→ 显存、并发、延迟、通信和成本约束
→ 排队、OOM、冷启动、资源碎片和租户争抢
→ 推理引擎、调度、网关、缓存、弹性和治理方案
→ 规格、配额、策略、监控、计量和SLA产品能力
→ 用户体验、资源效率和商业价值指标
```

### 三大生命周期到经营闭环

```text
资源生命周期提供可运行算力
→ 模型生命周期形成可信模型服务
→ 应用生命周期产生真实业务请求
→ 请求产生用量、质量、成本和业务结果
→ 计量、成本、价格、账单和毛利
→ 反向驱动容量、调度、套餐和产品迭代
```

## 5. 文档使用原则

1. 文档是决策工具，不是交付数量竞赛。
2. 行业判断必须继续通过客户访谈、工单和数据验证。
3. 没有真实证据的数字应写成“目标值”或“价值假设”。
4. 客户项目应在通用模板基础上形成独立版本，避免直接照搬。
5. PRD负责描述产品需求，HLD/LLD负责技术实现，SID负责系统接口。
6. 项目上线不是终点，必须通过指标和客户反馈验证产品价值。
7. 自动审计只能判断结构信号，不能替代产品、架构、安全和客户的正式评审。
8. 技术八股只能作为知识索引，最终必须回答适用场景、代价、边界、故障处置和产品化方式。
9. 产品经理应达到能判断和验证底层方案的深度，但不得虚构CUDA、NCCL或调度器研发经历。
