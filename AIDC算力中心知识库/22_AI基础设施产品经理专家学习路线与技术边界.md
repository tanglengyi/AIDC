## 1. 定位结论

AI基础设施产品经理需要达到“能理解约束、判断方案、设计产品、组织验证”的深度，而不是伪装成CUDA内核、通信库或调度器研发专家。

真正的专家能力分为四层：

1. **原理理解**：知道GPU、显存、通信、调度和推理为什么形成瓶颈。
2. **工程判断**：能识别OOM、通信退化、资源碎片、冷启动和长尾延迟的主要原因。
3. **产品化能力**：能把技术机制转化为规格、策略、配额、流程、告警、SLO和成本功能。
4. **业务经营**：能说明方案如何改善上线效率、资源利用率、稳定性、单位成本和毛利。

不要求产品经理独立实现CUDA Kernel、NCCL通信算法、推理引擎或Kubernetes调度插件，但必须知道这些组件解决什么问题、有什么边界、出现问题时该看什么证据。

## 3. CUDA与GPU：学到什么程度

### 3.1 必须理解的概念

- CPU擅长低延迟、复杂控制流；GPU通过大量并行线程和高内存带宽提升吞吐。
- Kernel是运行在GPU上的计算函数；Kernel启动、同步和数据搬运都有开销。
- GPU显存不只是“容量”，还涉及带宽、访问模式、缓存和数据搬运。
- 训练和推理都可能受算力、显存容量、显存带宽或通信限制，不能只看TFLOPS。
- 同一模型在不同精度、批大小、上下文和并行策略下，显存与性能完全不同。

### 3.2 产品场景

**场景：GPU利用率不高但推理仍然慢。**

可能原因：

- 请求在网关或队列中等待。
- Decode阶段受显存带宽限制。
- CPU Tokenization或数据预处理成为瓶颈。
- Kernel过碎、启动开销高。
- 多卡通信等待。

产品解法：端到端阶段指标、硬件画像、规格模板、压测报告和瓶颈解释。

### 3.3 学习边界

你应能解释Thread、Block、SM、Kernel、Global Memory、Shared Memory的大致关系；不要求现场手写矩阵乘Kernel。

### 3.4 学习资料

- NVIDIA CUDA Programming Guide: https://docs.nvidia.com/cuda/cuda-programming-guide/
- NVIDIA CUDA Best Practices Guide: https://docs.nvidia.com/cuda/cuda-c-best-practices-guide/
- NVIDIA Nsight Systems: https://docs.nvidia.com/nsight-systems/
- NVIDIA Nsight Compute: https://docs.nvidia.com/nsight-compute/

建议阅读顺序：Programming Model → Memory → Asynchronous Execution → Multi-GPU概览 → 使用Profiler理解时间线。

## 4. NCCL、网络与GPU拓扑

### 4.1 必须理解

NCCL是面向多GPU的集合通信库，常见操作包括AllReduce、AllGather、ReduceScatter和Broadcast。它会结合GPU和网络拓扑进行通信，但最终性能仍取决于NVLink/NVSwitch、PCIe、InfiniBand/RoCE、网卡绑定、跨NUMA和网络配置。

### 4.2 为什么产品经理要懂

- 多卡任务不是“卡数翻倍，速度就翻倍”。
- 通信占比过高时，增加GPU可能反而降低性价比。
- 同一节点内和跨节点并行的性能差异很大。
- GPU拓扑必须成为调度、规格和成本估算的一部分。

### 4.3 高频故障场景

**场景一：8卡训练比4卡快不了多少。**

检查：计算/通信占比、AllReduce耗时、跨节点比例、网络带宽、GPU拓扑、数据加载。

**场景二：NCCL初始化或通信挂起。**

检查：网卡选择、端口、防火墙、驱动/CUDA/NCCL兼容、节点间连通、Rank配置和异步错误。

### 4.4 产品能力

- 拓扑感知资源池和调度。
- 节点内/跨节点规格说明。
- 通信压测、带宽基线和异常告警。
- 训练任务时间线中区分计算、通信和数据等待。
- 对用户解释为什么某种多卡规格不具备线性加速。

- NVIDIA GPUDirect RDMA: https://docs.nvidia.com/cuda/gpudirect-rdma/

## 5. 分布式训练

### 5.1 需要理解的四种并行

| 方式 | 核心思想 | 适用问题 | 主要代价 |
|---|---|---|---|
| 数据并行 | 每张卡保存模型副本，处理不同数据 | 模型能放进单卡 | 梯度同步通信 |
| 张量并行 | 将单层矩阵计算切到多卡 | 单卡放不下大层或需要低延迟多卡推理 | 高频通信、拓扑敏感 |
| 流水线并行 | 不同层放在不同设备 | 模型过深、跨卡切层 | 流水线气泡和调度复杂 |
| 参数/状态分片 | 参数、梯度、优化器状态分散存储 | 训练显存不足 | 通信、重建和容错复杂 |

### 5.2 产品经理应能回答

- 为什么训练需要Checkpoint？因为任务周期长，节点或网络故障会导致大量算力浪费。
- 为什么需要Gang Scheduling？多卡任务必须同时获得完整资源，否则部分卡占着但任务无法开始。
- 为什么小任务会切碎大集群？零散分配可能破坏连续卡和高速拓扑。
- 为什么训练队列和在线推理队列应隔离？两者SLO、持续时间和容错方式不同。

### 5.3 平台产品能力

- 分布式任务模板和并行策略。
- Gang调度、队列、配额、优先级和预约。
- Checkpoint策略、对象存储和断点续训。
- 故障节点自动隔离与任务重试。
- 训练效率、通信占比和成本报告。

- PyTorch FSDP: https://docs.pytorch.org/docs/stable/fsdp.html

## 6. 推理引擎与LLM Serving

### 6.1 必须掌握的因果链

```text
模型权重大、自回归逐Token生成、请求长度不同、显存有限
→ 冷启动、KV Cache增长、吞吐与延迟冲突、请求互相阻塞
→ PagedAttention、Continuous Batching、Prefix Cache、量化、并行、弹性
→ 上下文/并发/Token限制、策略模板、SLO、成本和告警
```

### 6.2 关键概念

- Prefill：处理输入上下文，计算量通常较大。
- Decode：逐Token生成，频繁访问KV Cache，延迟敏感。
- TTFT：从请求到首Token的时间。
- TPOT：后续Token之间的平均时间。
- KV Cache：保存历史注意力Key/Value，减少重复计算但占用显存。
- Continuous Batching：请求完成后动态补入新请求，提高资源利用率。
- Prefix Cache：复用相同前缀计算结果。
- 量化：降低显存和计算成本，但可能影响质量或兼容性。

### 6.3 必会场景

**长对话OOM**：限制上下文和输出Token，设置并发与KV Cache配额，使用分页缓存、回收、摘要或降级。

**高吞吐但P99恶化**：批大小或等待窗口过大，需长短请求分流、SLO队列和参数模板。

**缩容到零后首请求超时**：权重缓存、镜像预拉取、最小副本和热点预热。

### 6.4 学习资料

- vLLM Documentation: https://docs.vllm.ai/
- vLLM Paper: https://arxiv.org/abs/2309.06180
- NVIDIA Triton Inference Server: https://docs.nvidia.com/deeplearning/triton-inference-server/
- NVIDIA TensorRT-LLM: https://docs.nvidia.com/tensorrt-llm/
- KServe: https://kserve.github.io/website/docs/
- Hugging Face Text Generation Inference: https://huggingface.co/docs/text-generation-inference/

实践建议：用同一模型改变并发、输入长度、输出长度、批策略和量化，记录TTFT、TPOT、吞吐、显存和单位Token成本。

## 7. Kubernetes调度与AI负载

### 7.1 必须理解

- Device Plugin让Kubernetes发现和分配设备，但传统扩展资源通常以整数设备为主。
- Scheduler通过过滤、评分和绑定选择节点。
- Requests决定调度，Limits决定运行时限制；GPU设备的管理方式与CPU不同。
- 节点标签、污点/容忍、亲和性、拓扑和队列共同影响调度。
- DRA为设备分类、声明和更灵活的资源分配提供机制。

### 7.2 产品经理需要识别的边界

不能简单说“原生K8s不支持GPU”，正确说法是：Kubernetes提供通用设备和调度基础，但并不天然理解模型大小、KV Cache、模型缓存状态、通信拓扑、业务SLO和成本，因此AI平台需要增强调度和产品化解释。

### 7.3 高频场景

**Pod长期Pending**：资源不足、亲和/污点冲突、配额不足、拓扑不满足、PVC或镜像问题。

**集群有空闲GPU但任务调不进去**：资源碎片、型号不匹配、跨节点拓扑、Gang未满足或配额限制。

**扩容了但延迟没下降**：模型加载慢、无GPU容量、扩缩容指标错误或请求已在网关排队。

## 8. 调度器算法：产品经理学什么

### 8.1 需要理解的结构

```text
任务进入队列
→ 准入与配额判断
→ 候选节点过滤
→ 节点评分
→ 资源预留
→ 绑定
→ 运行中监控
→ 抢占/迁移/回收
```

### 8.2 需要掌握的目标冲突

- 公平与业务优先级。
- 利用率与资源碎片。
- 数据局部性与负载均衡。
- 抢占效率与在途任务损失。
- 低延迟与低成本。
- 当前最优与未来大任务可调度性。

### 8.3 不要求的内容

不要求现场实现调度算法，但应能给出：调度输入、约束、目标函数、失败解释、可配置策略和验证指标。

### 8.4 推荐实践

设计一个4节点、每节点8卡的资源池，模拟1卡、2卡、4卡和8卡任务，比较FIFO、公平、回填和拓扑优先策略对排队时间、利用率和碎片率的影响。

## 9. 算子优化与推理性能

### 9.1 产品经理需要知道

算子是深度学习计算的基本单元。算子融合、FlashAttention、量化Kernel、CUDA Graph和编译优化可以减少内存访问、Kernel启动和中间结果开销。

产品层面不能承诺“开启优化一定更快”，因为效果受模型结构、批大小、序列长度、硬件、精度和引擎版本影响。

### 9.2 产品化方式

- 硬件—模型—引擎兼容矩阵。
- 优化开关和策略模板。
- 基线压测与版本对比。
- 质量回归和自动回滚。
- 性能收益、显存变化和成本变化报告。

## 10. 可观测性、故障与SRE

### 10.1 四类信号

- Metrics：连续数值趋势。
- Logs：具体行为和错误上下文。
- Traces：请求经过应用、网关、路由和推理实例的链路。
- Events：部署、扩缩容、OOM、驱动错误、策略变更等状态变化。

### 10.2 AI平台特有指标

GPU利用率、显存、功耗、温度、ECC、KV Cache、排队、并发、TTFT、TPOT、吞吐、上下文分布、模型加载、缓存命中、Token和单位成本。

### 10.3 故障分析模板

```text
症状
→ 影响的租户/应用/模型
→ 最近变更
→ 网关/队列/推理/资源分段耗时
→ 资源和拓扑状态
→ 止损动作
→ 根因证据
→ 恢复验证
→ 防复发措施
```

## 11. 成本、计费和FinOps

### 11.1 必须掌握

- 资源利用率不等于有效产出。
- 已分配成本、实际使用成本和空闲成本需要分开。
- GPU卡时、实例时长和Token计费适用于不同交付对象。
- 共享资源必须有明确分摊规则。
- 每百万Token成本要结合模型、硬件、输入输出比例、缓存命中和服务等级计算。

### 11.2 产品功能

- 应用/模型/租户级Showback和Chargeback。
- 预算、预测、异常消耗和超额策略。
- 价格版本、套餐、账单、冲正和对账。
- 单卡收入、单位Token毛利和容量采购建议。

### 11.3 学习资料

- FinOps Framework: https://www.finops.org/framework/
- OpenCost: https://www.opencost.io/docs/

## 12. 12周学习路线

### 第1—2周：GPU与系统基础

目标：讲清CPU/GPU差异、显存、带宽、Kernel、精度和OOM。

产出：一份“模型显存和部署规格估算表”。

### 第3周：CUDA时间线与性能分析

目标：能看懂CPU、Kernel、Memcpy和同步时间线。

产出：一个简单GPU程序或现成模型的Nsight时间线解读。

### 第4周：NCCL与拓扑

目标：讲清AllReduce、NVLink/PCIe/RDMA和跨节点代价。

产出：单机多卡与跨机多卡性能对比说明。

### 第5周：分布式训练

目标：理解DP、TP、PP、FSDP/ZeRO和Checkpoint。

产出：不同模型规模的并行方案选型表。

### 第6—7周：LLM推理

目标：掌握Prefill、Decode、KV Cache、Batching、量化和并行。

产出：vLLM压测报告，包含TTFT、TPOT、吞吐和显存。

### 第8周：Kubernetes AI调度

目标：理解Device Plugin、Scheduler、DRA、队列和Gang。

产出：GPU任务Pending故障树和调度失败解释页面原型。

### 第9周：模型生命周期与LLMOps

目标：建立评测、准入、部署、灰度、回滚和下线闭环。

产出：模型上线PRD和状态机。

### 第10周：多租户与安全

目标：设计租户、项目、角色、Key、网络和审计。

产出：权限矩阵与数据隔离方案。

### 第11周：成本与计费

目标：建立用量、成本、价格、账单和毛利模型。

产出：Token与卡时混合计费设计。

### 第12周：综合作品集和面试

目标：用业务问题—约束—方案—代价—指标—价值讲完整项目。

产出：5分钟、15分钟和深挖版三套讲稿。

## 14. 专家级面试判断标准

能够做到以下几点，已足以胜任大多数MaaS/AI基础设施产品岗：

1. 面对技术名词，可以从系统约束推导而不是背定义。
2. 面对故障，可以提出分层假设、所需证据和止损方案。
3. 面对方案，可以说明收益、代价、边界和替代方案。
4. 面对产品设计，可以给出对象、状态、规则、权限、流程、指标和验收。
5. 面对商业问题，可以把GPU、Token和SLO转化为成本、价格、预算和毛利。
6. 明确知道自己不是底层内核研发，不编造生产事故和实现细节。
