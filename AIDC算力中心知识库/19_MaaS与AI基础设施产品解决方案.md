版本：V1.0  
日期：2026-07-20  
用途：产品规划、方案评审、售前交流、PRD拆解、面试作品集  
适用岗位：MaaS产品经理、AI平台产品经理、AIGC基础设施产品经理、智算平台产品经理、云原生产品经理

## 1. 方案结论

MaaS与AI基础设施平台不是“模型上传加一键部署”的功能集合，而是围绕模型在企业生产环境中运行时的四类核心矛盾建立的一套产品体系：

1. **资源昂贵且异构**：GPU/NPU型号、显存、拓扑和软件栈不同，资源总量充足也可能无法调度。
2. **推理负载不同于普通Web服务**：模型权重大、冷启动慢、KV Cache持续增长、请求长度差异大，吞吐与延迟存在天然冲突。
3. **企业使用需要治理**：多租户共享算力时必须解决权限、配额、限流、审计、数据隔离和服务等级。
4. **平台必须形成经营闭环**：不仅要部署模型，还要能够计量、计费、成本归因、容量规划和持续运营。

因此，完整解决方案应形成以下闭环：

```text
算力纳管
  → 资源池化
  → 模型资产管理
  → 评测与准入
  → 部署与推理
  → 流量与队列治理
  → 可观测与故障处置
  → 用量计量与成本归因
  → 运营分析与容量规划
  → 模型下线与资源回收
```

平台价值不是“拥有多少技术组件”，而是能否证明以下结果：

- 模型从申请到上线的时间缩短。
- GPU有效利用率提升，空闲与资源碎片降低。
- 在线推理延迟、失败率和OOM率受控。
- 多租户之间不会相互拖垮。
- 每个模型、项目、租户的成本和收入可以追溯。
- 故障可以快速定位、止损、恢复并复盘。

## 2. 产品定位与边界

### 2.1 产品定位

平台定位为连接企业AI应用、模型资产、推理引擎、Kubernetes/Slurm集群和GPU/NPU资源池的AI服务生产与运营中枢。

它既不是单一的模型仓库，也不是单一的Kubernetes控制台，更不是只提供聊天界面的AIGC应用。其核心职责是把底层算力与推理机制转化为可申请、可部署、可治理、可计量、可运营的企业级模型服务。

### 2.2 与相邻平台的关系

| 平台类型 | 核心职责 | 与本方案关系 |
|---|---|---|
| IaaS/云管平台 | 云主机、裸金属、网络、存储、资源交付 | 提供基础资源，MaaS向上封装模型服务 |
| Kubernetes平台 | 容器编排、服务发现、基础调度 | 作为通用运行底座，需要AI负载增强 |
| Slurm/HPC平台 | 批处理、训练、科研作业和队列 | 承载训练与离线计算任务 |
| MLOps平台 | 数据、实验、训练、评测、模型版本 | 向MaaS交付已准入模型 |
| MaaS平台 | 模型部署、推理API、流量治理、Token计量 | 本方案核心范围 |
| Agent/RAG平台 | 知识库、工作流、工具调用、智能体应用 | MaaS的上层消费方 |
| API网关/服务网格 | 鉴权、路由、限流、熔断、mTLS | 被集成用于流量和服务治理 |
| DCIM/AIDC运营平台 | 机房、能耗、资产、算力经营 | 提供设施和资源运营数据 |

### 2.3 不应混为一谈的三类交付

1. **租GPU**：交付GPU云主机、裸金属或容器资源，通常按卡时或包月计费。
2. **交付模型实例**：客户选择模型和规格，平台提供专属或共享推理实例。
3. **交付模型API**：平台隐藏底层资源，通过统一API提供模型能力，通常按Token、请求量或套餐计费。

产品经理必须先判断客户购买的是哪一种交付，才能确定一期范围、SLA和计费方式。

## 4. 核心痛点、根因与产品解法

### 4.1 痛点一：资源总量有剩余，但任务调不进去

**典型场景**：集群还有多张空闲GPU，但某个需要4卡高速互联的模型持续排队。

**根因**：

- 空闲卡分散在不同节点。
- GPU型号、显存或驱动版本不一致。
- 节点之间缺少所需高速互联。
- 租户配额、节点亲和性和污点策略冲突。
- 资源被小任务切碎，形成不可用碎片。

**产品解法**：

- 建立GPU资源画像：型号、显存、健康、拓扑、驱动、互联和可分配状态。
- 支持队列、项目配额、优先级、预约和Gang Scheduling。
- 提供调度失败原因解释，而不是只显示Pending。
- 提供资源碎片率、可调度容量和整理建议。
- 支持MIG、整卡、时间切分等不同共享方式，并明确隔离等级。

**核心指标**：调度成功率、平均排队时长、资源碎片率、GPU有效利用率。

### 4.2 痛点二：模型部署成功，但首个请求等待数分钟

**典型场景**：Serverless实例缩容到零后，客户第一次调用出现超时。

**根因**：

```text
Pod创建
→ 镜像拉取
→ 模型权重下载
→ 权重加载到内存
→ 权重加载到GPU显存
→ 推理引擎初始化
→ 算子或图编译
→ 预热请求
```

**产品解法**：

- 模型权重本地缓存和分布式缓存。
- 镜像预拉取与专用预热节点池。
- 最小副本数、定时预热、热点模型常驻。
- 冷启动过程分阶段展示，便于定位卡点。
- 将“成本优先”和“低延迟优先”做成可选服务等级。

**核心指标**：部署时长、冷启动时长、模型加载时长、首Token延迟、缓存命中率。

### 4.3 痛点三：长对话和高并发导致显存持续上涨并OOM

**根因**：模型权重长期占据显存，KV Cache又随上下文长度、并发数和生成长度增长。长请求与大量并发请求会争抢剩余显存。

**产品解法**：

- 模型级最大上下文长度和最大输出Token。
- 租户级并发、Token速率和KV Cache配额。
- Paged KV Cache、Prefix Cache、缓存回收与必要时的分层卸载。
- 超长会话摘要、截断、排队、拒绝或降级策略。
- 显存水位、KV Cache使用率和OOM风险告警。

**核心指标**：显存利用率、KV Cache使用率、OOM次数、缓存命中率、上下文长度分布。

### 4.4 痛点四：GPU利用率低，但请求延迟仍然很高

**典型场景**：监控显示GPU利用率只有30%，客户却反馈首Token慢。

**可能原因**：

- 请求大部分时间在网关或队列等待。
- 模型加载、数据预处理或Tokenization成为瓶颈。
- 请求长度差异大，长请求阻塞短请求。
- 批处理等待窗口过长。
- 多卡通信、网络或存储拖慢。
- 指标只看瞬时计算利用率，未反映显存带宽或Decode阶段特征。

**产品解法**：

- 将端到端耗时拆成网关、排队、Prefill、Decode和传输阶段。
- 支持动态批处理和连续批处理参数管理。
- 支持长短请求分流、优先级和SLO队列。
- 依据等待请求数、KV Cache水位和并发数进行扩缩容，而不是只看CPU。
- 提供吞吐优先、延迟优先和成本优先的策略模板。

**核心指标**：TTFT、TPOT、端到端P95/P99、排队时长、输入/输出Tokens per Second。

### 4.5 痛点五：一个租户的批量任务拖垮在线服务

**根因**：在线对话、批量推理和微调任务共享同一资源池和队列，缺乏优先级、隔离和限流。

**产品解法**：

- 在线推理、批量推理、训练/微调分队列和分资源池。
- 项目与租户配额、并发上限、Token速率限制和预算限制。
- 支持优先级、抢占、预约和借用空闲配额。
- 过载时执行排队、快速失败、降级模型或转移到备用集群。

**核心指标**：SLO违约率、队列等待、租户资源占用、限流次数、抢占次数。

### 4.6 痛点六：原生Kubernetes能运行Pod，却不能充分理解大模型负载

**原因**：Kubernetes提供通用编排和设备分配能力，但普通Deployment和默认调度策略不会自动理解模型大小、权重缓存、KV Cache、GPU拓扑、请求SLO及不同任务的业务优先级。

**产品解法**：

- 通过Device Plugin或适配的动态资源分配机制纳管加速设备。
- 增强调度器或调度插件：拓扑感知、模型缓存感知、队列、Gang和优先级。
- 通过KServe、模型控制器或自研Operator管理推理服务生命周期。
- 通过推理引擎指标驱动扩缩容。
- 将技术参数封装成产品规格，避免业务用户直接配置复杂YAML。

### 4.7 痛点七：模型版本上线后，基础设施正常但业务效果下降

**根因**：传统灰度只观察CPU、错误率和延迟，而模型升级还可能改变回答质量、安全性、Token消耗和业务转化。

**产品解法**：

- 模型版本、Prompt版本、推理参数和评测集一起版本化。
- 上线前执行离线质量、安全、性能和成本评测。
- 上线时按租户、API Key、请求标签或流量比例灰度。
- 同时监控技术指标、模型质量指标和业务指标。
- 设定自动回滚条件，并保存回滚证据。

### 4.8 痛点八：平台有监控，但故障发生时仍不知道谁受影响

**根因**：GPU、节点、Pod、模型、API Key、租户和业务应用的数据模型相互割裂。

**产品解法**：

- 建立“设施—服务器—GPU—节点—Pod—推理实例—模型版本—API—租户—应用”拓扑。
- 统一指标、日志、Trace、事件和审计数据。
- 告警聚合、根因候选、影响面分析和Runbook联动。
- 故障工单自动携带相关版本、配置和时间线。

### 4.9 痛点九：GPU成本很高，但无法解释钱花在哪里

**根因**：共享实例、空闲预热、KV Cache、失败请求、平台公共服务和跨租户复用使成本难以直接按请求分摊。

**产品解法**：

- 采集卡时、实例时长、输入/输出Token、存储、流量和公共服务成本。
- 支持模型、租户、项目、应用、API Key多维成本归因。
- 区分资源成本、平台成本、能耗成本和人工服务成本。
- 提供预算、预测、异常消耗告警和单位成本趋势。

### 4.10 痛点十：服务网格功能强，但引入后系统更复杂

**判断原则**：

- 南北向模型API治理优先由API网关或推理网关承担。
- 东西向服务数量多、调用复杂、需要统一mTLS、可观测和细粒度流量策略时，再评估服务网格。
- 小集群、服务少、团队运维能力不足时，不应为了“云原生完整”而强行引入。

**产品解法**：

- 网关和网格能力分层，不把所有治理都绑定到一个组件。
- 提供策略模板、灰度开关、超时/重试冲突检查和拓扑视图。
- 评估Sidecar资源开销和运维复杂度；必要时选择更轻量的数据平面模式。

## 6. 产品能力地图

### 6.1 算力资源与集群管理

- 多集群、多资源池和多可用区纳管。
- GPU/NPU型号、显存、拓扑、健康和驱动画像。
- 整卡、MIG、时间切分或厂商虚拟化资源规格。
- 节点池、标签、污点、亲和性和资源预约。
- Kubernetes与Slurm统一视图。
- 资源配额、借用、回收、超卖策略和碎片分析。

### 6.2 模型资产与准入

- 模型、版本、权重、格式、许可证和来源管理。
- 模型依赖、运行环境、镜像和硬件兼容矩阵。
- 质量、安全、性能、成本四类评测。
- 准入审批、黑白名单和生命周期状态。
- 模型卡、变更记录、责任人和下线策略。

### 6.3 模型部署与推理服务

- 在线推理、批量推理、专属实例和共享实例。
- 服务规格：模型、精度、上下文、GPU类型、副本和服务等级。
- 部署模板、健康检查、预热、灰度和自动回滚。
- 多卡并行、跨节点部署和拓扑校验。
- 模型缓存、镜像缓存和启动阶段可视化。

### 6.4 推理优化策略

- 动态批处理与连续批处理。
- KV Cache、Paged KV Cache和Prefix Cache。
- 量化、推测解码和并行策略配置。
- 长短请求分流、Prefill/Decode指标和优先级。
- 最大上下文、最大输出、最大并发和等待窗口。

产品页面不应直接暴露所有底层参数，应提供三层配置：

1. 业务模板：低延迟、均衡、低成本。
2. 高级参数：并发、批大小、上下文、缓存和扩缩容。
3. 专家模式：推理引擎原生参数。

### 6.5 API与流量治理

- 统一OpenAI兼容或企业标准API。
- API Key、OAuth2/OIDC、签名、IP和网络访问控制。
- 按租户、应用、模型和Key限流。
- Token速率、并发数、请求数和预算限额。
- 模型路由、故障切换、灰度和A/B测试。
- 超时、重试、熔断、降级、排队和快速失败。
- 请求与响应日志脱敏和采样。

### 6.6 弹性与队列

- 最小/最大副本、定时扩缩容、热点预热。
- 基于等待请求、运行请求、KV Cache水位、延迟和自定义指标扩缩。
- 在线、批量、微调分队列。
- 优先级、抢占、预约、配额借用和公平调度。
- 扩容失败、无资源和排队超时的用户可解释提示。

### 6.7 可观测与智能运维

- 指标：GPU、显存、KV Cache、请求、队列、TTFT、TPOT、吞吐、成本。
- 日志：平台日志、引擎日志、模型日志、网关日志和审计日志。
- Trace：应用—网关—路由—推理实例的端到端链路。
- 事件：Pod重启、OOM、掉卡、模型加载、扩缩容和策略变更。
- 拓扑：资源、模型、服务、租户和应用关联。
- 告警聚合、影响面、根因候选、Runbook和复盘。

### 6.8 多租户、权限与安全

- 租户—项目—用户—角色—资源的分层模型。
- 平台角色、租户角色和项目角色分离。
- 模型、数据集、API、日志和账单的数据隔离。
- Secret和API Key托管、轮换、吊销和过期。
- 模型权重、镜像和依赖安全扫描。
- 操作、调用、下载和配置变更审计。
- 私有化、专属集群、网络隔离和数据不出域方案。

### 6.9 计量、计费与成本运营

支持以下计量对象：

- GPU卡时、实例时长和预留资源。
- 输入Token、输出Token、缓存Token和请求次数。
- 存储容量、模型权重、日志和网络流量。
- 专属集群、SLA等级和人工运维服务。

支持以下商业模式：

- 包年包月。
- 按量计费。
- 预付费套餐和额度包。
- 专属实例保底加超量。
- 免费额度加阶梯计价。
- 内部成本中心分摊。

## 9. 服务等级与指标体系

### 9.1 用户体验指标

- 模型申请到可调用时间。
- 部署成功率。
- TTFT首Token延迟。
- TPOT单Token生成延迟。
- P50/P95/P99端到端延迟。
- 请求成功率和流式中断率。
- API可用率。

### 9.2 资源效率指标

- GPU有效利用率。
- 显存利用率和KV Cache使用率。
- 资源碎片率。
- 队列平均与P95等待时长。
- 单实例吞吐和每卡吞吐。
- 模型缓存命中率。
- 冷启动与预热资源占比。

### 9.3 稳定性指标

- OOM率。
- 扩缩容失败率。
- 模型加载失败率。
- MTTA、MTTR和重复故障率。
- 灰度回滚次数。
- SLO违约次数和错误预算消耗。

### 9.4 商业指标

- 每百万Token成本。
- 每百万Token收入和毛利。
- 单卡收入、卡时成本和空闲成本。
- 租户预算消耗率。
- 套餐续费率和客户留存。
- 可售资源、已售资源和预留资源比例。

### 9.5 不应单独使用的指标

- 只看GPU利用率：不能说明请求是否在排队，也不能说明用户延迟。
- 只看QPS：不同请求Token长度差异极大。
- 只看平均延迟：可能掩盖P99长尾。
- 只看Token数量：不能反映模型规格和资源成本差异。
- 只看功能上线率：不能证明客户价值。

## 10. 关键产品决策与取舍

| 决策 | 方案A | 方案B | 选择依据 |
|---|---|---|---|
| 资源交付 | 专属实例 | 共享实例 | 隔离、成本、流量稳定性 |
| 弹性策略 | 最小副本常驻 | 缩容到零 | 延迟SLO与空闲成本 |
| 推理目标 | 低延迟 | 高吞吐 | 在线交互或批量任务 |
| GPU使用 | 整卡 | 切分/共享 | 性能隔离和模型大小 |
| 网格选型 | 仅API网关 | 服务网格 | 东西向复杂度、mTLS和团队能力 |
| 缓存策略 | 高缓存占用 | 激进回收 | 命中率、显存和并发 |
| 模型版本 | 单版本 | 多版本灰度 | 业务风险和资源成本 |
| 计费方式 | 卡时 | Token | 交付对象是资源还是API |
| 调度目标 | 公平 | 业务优先级 | 公共平台或关键业务保障 |
| 扩缩容指标 | CPU/GPU | 请求/KV/延迟 | 是否能反映真实推理负载 |

产品经理在评审时必须说明：选择解决了什么问题、牺牲了什么、适用边界是什么、用什么指标验证。

## 12. PRD级验收示例

### 12.1 上下文和并发保护

**需求**：管理员可以为模型服务设置最大上下文、最大输出Token、租户并发和Token速率。

**验收**：

1. 超过上下文限制的请求按配置执行拒绝、截断或摘要。
2. 超过并发限制时执行排队或快速失败，并返回可识别错误码。
3. 页面显示当前额度、已用额度和预计等待时间。
4. 所有超限事件可按租户、模型和API Key查询。
5. 压测期间不得因单租户超限导致其他租户SLO显著恶化。

### 12.2 模型灰度与回滚

**需求**：支持按流量比例、租户或API Key将请求路由到新模型版本。

**验收**：

1. 灰度比例可配置并可实时停止。
2. 新旧版本分别展示延迟、错误、质量、Token和成本指标。
3. 达到回滚阈值时自动停止新版本流量。
4. 回滚后保留完整时间线、配置和指标快照。

### 12.3 推理弹性

**需求**：支持基于等待请求数、运行请求数和KV Cache使用率扩缩容。

**验收**：

1. 管理员可配置最小/最大副本和扩缩容阈值。
2. 扩容过程展示资源申请、Pod启动、模型加载和就绪阶段。
3. 无足够GPU时提供明确原因和建议，而不是无限等待。
4. 扩容后P95排队时间应回落到目标区间。
5. 缩容不得中断正在生成的流式请求。

## 13. 面试作品集讲法

建议使用以下五分钟结构：

### 13.1 业务问题

企业已经采购GPU并部署Kubernetes，但模型上线慢、资源利用率低、多租户互相影响，且成本无法归因。

### 13.2 根因判断

问题不只是缺少一个部署页面，而是通用容器编排无法自动理解模型权重、KV Cache、请求SLO和业务优先级，模型服务也缺少从准入、流量治理到计量运营的完整闭环。

### 13.3 产品方案

建设资源池、模型仓库、推理服务、API治理、队列配额、可观测、成本计量和运营分析八大能力，并通过统一数据模型贯通GPU、实例、模型、租户和应用。

### 13.4 关键取舍

- 在线与批量任务分队列。
- 扩缩容不只看CPU，增加等待请求和KV Cache指标。
- 南北向流量先用推理网关，东西向复杂后再引入服务网格。
- 低延迟模型保留最小副本，低频模型允许缩容到零。

### 13.5 成功指标

模型上线时长、TTFT、P95排队、OOM率、GPU有效利用率、每百万Token成本和租户SLO达成率。

## 14. 面试回答总公式

所有模块都可以按以下逻辑回答：

```text
技术或硬件特性
→ 系统约束
→ 用户/运维/成本痛点
→ 技术方案
→ 新增代价与适用边界
→ 产品控制能力
→ 验证指标与商业价值
```

例如连续批处理：

> 大模型请求长度不同，传统静态批次必须等待整个批次结束，短请求完成后对应GPU槽位会空闲。连续批处理允许请求完成后动态补入新请求，提高吞吐和GPU利用率。但更激进的批处理会增加排队和显存压力，因此产品侧需要提供最大并发、等待窗口、优先级和显存水位保护，并同时观察TTFT、TPOT、吞吐、KV Cache占用和每百万Token成本。
