# MaaS角色用户旅程与资源—模型—应用生命周期

版本：V1.0  
日期：2026-07-20  
用途：MaaS/AI基础设施产品规划、PRD拆解、用户旅程设计、状态机设计、计量运营与面试作品集

---

## 1. 文档目标

本文件补齐MaaS与AI基础设施产品中容易被技术模块掩盖的三条产品主线：

1. 谁购买、谁管理、谁开发、谁使用、谁运维，以及各角色从进入平台到退出平台的完整旅程。
2. 资源、模型、应用分别经历哪些状态，三条生命周期如何互相约束和联动。
3. 配额如何逐层分配、用量如何采集、成本如何归因、价格如何形成、账单如何结算、指标如何反向驱动容量与产品决策。

平台不是若干页面的集合，而是一组相互关联的对象、状态、规则和运营闭环。

## 2. 角色地图与责任边界

### 2.1 购买与决策角色

| 角色 | 关心的问题 | 主要决策 | 产品触点 |
|---|---|---|---|
| 企业管理者/业务负责人 | AI能力是否产生业务价值，风险和成本是否可控 | 是否采购、预算规模、SLA等级 | 服务目录、经营驾驶舱、业务效果报表 |
| IT/平台负责人 | 平台能否统一治理，能否与现有系统集成 | 架构选型、部署方式、自研与采购边界 | 总体架构、集成中心、容量与成本报表 |
| 财务/采购 | 价格是否透明，账单能否核对 | 合同、套餐、结算方式 | 报价、账单、对账、预算与成本中心 |
| 安全与合规负责人 | 数据、模型和接口是否安全可审计 | 安全基线、数据边界、准入规则 | 权限、审计、扫描、密钥、合规报告 |

### 2.2 管理与运营角色

| 角色 | 核心职责 | 典型权限 |
|---|---|---|
| 平台超级管理员 | 集群、资源池、全局策略和租户管理 | 全局配置、资源纳管、策略发布、审计查看 |
| 租户管理员 | 本组织项目、成员、配额和预算管理 | 创建项目、分配配额、审批申请、查看本租户账单 |
| 项目管理员 | 单个项目内模型、应用和成员协作 | 模型准入、应用授权、预算与告警配置 |
| 运营管理员 | 套餐、价格、账单、活动和经营分析 | 产品上架、价格配置、优惠、账单调整审批 |
| SRE/运维 | 稳定性、故障处置、容量和变更 | 监控、告警、工单、Runbook、实例操作 |
| 安全管理员 | 身份、密钥、镜像、模型和审计治理 | 安全策略、Key吊销、扫描结果和审计访问 |

### 2.3 生产与消费角色

| 角色 | 核心任务 | 关键成功指标 |
|---|---|---|
| 算法工程师 | 上传、评测、微调、发布模型 | 模型上线时长、评测通过率、回滚率 |
| AI应用开发者 | 获取Key、调用API、调试并发布应用 | 首次调用成功率、接入时长、错误定位时长 |
| 数据工程师 | 准备数据集、数据权限与数据质量 | 数据可用率、数据处理时长、合规通过率 |
| 业务用户 | 在业务系统中消费AI能力 | 任务完成率、响应时延、满意度、业务转化 |

## 3. 统一用户旅程框架

每条旅程均按以下结构描述：

```text
触发事件 → 用户目标 → 用户动作 → 平台触点 → 风险/阻塞 → 产品响应 → 成功指标
```

### 3.1 算法工程师旅程

| 阶段 | 用户动作 | 典型痛点 | 平台能力 | 指标 |
|---|---|---|---|---|
| 发现 | 查询可用算力、基础模型和模板 | 不知道硬件兼容性和成本 | 模型目录、硬件兼容矩阵、成本预估 | 查询到提交转化率 |
| 准备 | 上传权重、镜像、许可证和模型卡 | 文件大、依赖不一致、来源不清 | 断点上传、制品仓库、依赖扫描、模型卡模板 | 上传成功率、准备时长 |
| 评测 | 提交质量、安全、性能和成本评测 | 指标不统一、结果不可比 | 标准评测集、基线版本、评测报告 | 评测完成时长、通过率 |
| 准入 | 提交审批并选择规格 | 审批链不清、规格选择困难 | 准入工作流、推荐规格、成本上限 | 审批时长、退回率 |
| 发布 | 部署预发布并灰度 | 冷启动慢、问题难回滚 | 预热、灰度、自动回滚、变更时间线 | 部署成功率、回滚率 |
| 运营 | 观察质量、延迟、吞吐和成本 | 技术指标与业务效果割裂 | 多维看板、版本对比、异常告警 | SLO、单位成本、质量漂移 |
| 迭代/退出 | 发布新版本、归档或下线 | 依赖关系不清，误下线 | 影响分析、双版本切换、归档和回收 | 无损升级率、资源回收时长 |

### 3.2 AI应用开发者旅程

| 阶段 | 用户动作 | 典型痛点 | 平台能力 | 指标 |
|---|---|---|---|---|
| 选型 | 比较模型效果、价格和SLA | 模型名称多但不可比较 | 模型卡、在线试用、统一基准、费用模拟 | 试用转申请率 |
| 申请 | 创建应用、申请模型和额度 | 预算与审批不透明 | 应用注册、预算审批、配额说明 | 平均开通时长 |
| 接入 | 创建API Key、下载SDK和联调 | 鉴权、错误码和参数复杂 | 开发者中心、兼容API、示例代码、沙箱 | 首次调用成功率 |
| 测试 | 压测、验证错误和限流策略 | 测试环境与生产不一致 | 测试套餐、压测报告、SLO模拟 | 测试通过率 |
| 发布 | 将应用流量切到生产模型 | 发布风险、版本不可控 | 应用级路由、灰度、Key分环境 | 发布成功率 |
| 运营 | 查看用量、错误、延迟和预算 | 账单只能到租户，无法归因到应用 | 应用级用量、成本、质量和告警 | 应用单位调用成本 |
| 迁移/下线 | 更换模型、吊销Key、归档应用 | 害怕中断业务和凭证泄漏 | 双路由、兼容性检查、Key轮换和吊销 | 无损迁移率 |

### 3.3 租户管理员旅程

```text
租户开通
→ 建立组织和项目
→ 邀请成员并分配角色
→ 申请总配额和预算
→ 将额度分配到项目/应用
→ 审批模型和资源申请
→ 监控使用、成本和SLA
→ 处理超额、异常和账单
→ 续费、扩容、降配或注销
```

关键产品要求：

- 配额总量、已分配、已使用、可借用、被冻结必须区分。
- 预算、技术配额和商业套餐不能混为同一个字段。
- 管理员必须知道“为什么不能申请”，而不只是看到额度不足。
- 注销租户前必须完成资源释放、Key吊销、账单结清和审计归档。

### 3.4 SRE/运维旅程

```text
建立监控基线
→ 配置告警与值班
→ 发现异常
→ 关联资源、模型、应用和租户
→ 止损
→ 根因定位
→ 执行Runbook
→ 恢复验证
→ 复盘
→ 更新规则与容量计划
```

关键指标：MTTA、MTTR、告警压缩率、自动处置成功率、重复故障率、错误预算消耗。

### 3.5 财务/运营旅程

```text
定义产品与计量单位
→ 配置成本与价格
→ 建立套餐和优惠
→ 采集用量
→ 生成预估费用
→ 账单冻结
→ 对账与调整
→ 开票/内部结算
→ 毛利分析
→ 调价与容量采购
```

## 4. 资源生命周期

### 4.1 资源对象

资源不只是GPU卡，还包括：

- 设施：机房、机柜、供电、制冷。
- 节点：裸金属、虚拟机、Kubernetes Node。
- 加速器：GPU/NPU、整卡、MIG切片、时间共享实例。
- 网络与存储：高速网卡、交换网络、本地盘、并行文件系统、对象存储。
- 软件环境：驱动、CUDA/ROCm、固件、容器运行时和镜像缓存。

### 4.2 状态机

```text
待发现
→ 已发现
→ 待纳管
→ 初始化中
→ 健康检查中
→ 可用
→ 已预留
→ 已分配
→ 使用中
→ 释放中
→ 可用
```

异常分支：

```text
任意运行状态
→ 降级
→ 隔离
→ 维护中
→ 验证中
→ 可用 / 报废
```

### 4.3 每个阶段的产品要求

| 阶段 | 关键动作 | 风险 | 产品能力 |
|---|---|---|---|
| 发现 | 自动扫描BMC、节点和设备 | 重复资产、信息不完整 | 唯一标识、发现任务、冲突处理 |
| 纳管 | 绑定集群、资源池和责任人 | 权属不清 | 标签、组织、租户和成本中心 |
| 初始化 | 驱动、固件、运行时校验 | 软件栈不兼容 | 兼容矩阵、基线检查、自动修复 |
| 健康检查 | 压测显存、互联和ECC | 假健康资源进入生产 | 健康评分、准入阈值 |
| 规格化 | 生成整卡、MIG或共享规格 | 规格过多、隔离能力不透明 | 产品规格、性能等级、共享说明 |
| 预留/分配 | 队列、配额、预约和调度 | 碎片和抢占冲突 | 可调度容量、失败解释、Gang调度 |
| 使用 | 承载训练或推理任务 | 热点、超卖、性能干扰 | 监控、SLO、拓扑和隔离 |
| 释放 | 停止任务、清理显存和凭证 | 资源泄漏、数据残留 | 优雅终止、数据清理、回收确认 |
| 维护 | 升级驱动或维修硬件 | 影响在途任务 | 排空、迁移、维护窗、回滚 |
| 报废 | 退出资源池和资产系统 | 账实不符 | 审批、数据擦除、资产闭环 |

### 4.4 资源生命周期指标

- 纳管成功率和平均纳管时长。
- 健康资源占比、降级率、故障率。
- 可调度容量、已分配容量和碎片率。
- 分配等待时长和调度失败率。
- 资源泄漏率和平均回收时长。
- 每卡有效产出、空闲成本和维修成本。

## 5. 模型生命周期

### 5.1 状态机

```text
草稿
→ 上传中
→ 待校验
→ 待评测
→ 评测中
→ 待准入
→ 已准入
→ 待部署
→ 预发布
→ 灰度中
→ 生产中
→ 已停用
→ 已归档
→ 已删除
```

异常状态：校验失败、评测失败、部署失败、回滚中、隔离中。

### 5.2 生命周期控制点

| 阶段 | 必须校验的内容 |
|---|---|
| 上传 | 文件完整性、权重格式、来源、许可证、责任人 |
| 校验 | 镜像、依赖、硬件、框架、Tokenizer和配置兼容性 |
| 评测 | 质量、安全、性能、显存、吞吐、延迟和成本 |
| 准入 | 业务用途、数据级别、风险等级、预算和SLA |
| 部署 | 资源规格、并行方式、上下文、量化、缓存和弹性 |
| 灰度 | 新旧版本质量、错误、延迟、Token和成本对比 |
| 运营 | 漂移、异常输出、SLO、单位成本和应用影响 |
| 下线 | 依赖应用、路由、Key、缓存、账单和审计归档 |

### 5.3 模型版本关系

平台至少需要区分：

- 基础模型版本。
- 微调/LoRA适配版本。
- 量化和推理优化版本。
- 部署配置版本。
- API公开版本。

模型版本和部署版本不能混为一谈：同一个模型权重可以通过不同量化、并行、上下文和推理引擎形成多个服务规格。

## 6. 应用生命周期

### 6.1 应用对象定义

应用是消耗模型服务并产生业务流量、用量、成本和效果数据的最小运营单元。它需要绑定：

- 所属租户和项目。
- 负责人和成员。
- 环境：开发、测试、生产。
- 调用的模型服务和路由策略。
- API Key或身份凭证。
- 配额、预算、SLO和告警规则。
- 业务标签、成本中心和数据等级。

### 6.2 状态机

```text
草稿
→ 待审批
→ 开发中
→ 测试中
→ 待发布
→ 灰度中
→ 生产运行
→ 暂停
→ 下线中
→ 已归档
```

### 6.3 应用生命周期控制点

| 阶段 | 产品控制 |
|---|---|
| 创建 | 唯一名称、负责人、项目、用途和数据等级 |
| 申请 | 模型权限、预算、Token、并发和网络访问范围 |
| 开发 | 沙箱Key、SDK、示例、错误追踪和低额度保护 |
| 测试 | 压测、内容安全、故障降级、预算模拟 |
| 发布 | 生产Key、灰度比例、SLO、告警和值班责任人 |
| 运行 | 应用级成本、质量、延迟、错误和版本依赖 |
| 变更 | 模型切换、Key轮换、配额调整和审批记录 |
| 下线 | 停流、Key吊销、依赖检查、账单结清和归档 |

## 7. 三大生命周期的关联关系

```text
资源生命周期提供“可运行的算力”
        ↓
模型生命周期形成“可服务的模型版本”
        ↓
应用生命周期产生“真实业务请求”
        ↓
请求产生Token、卡时、流量、日志、质量与业务结果
        ↓
计量 → 成本 → 价格 → 账单 → 毛利 → 容量与产品决策
```

### 7.1 关键依赖规则

1. 资源进入“可用”前，模型部署不得选择该资源。
2. 模型未“已准入”，不得发布到生产环境。
3. 模型版本下线前，必须检查所有应用依赖。
4. 应用暂停后，应停止新的请求，但保留必要审计和账单数据。
5. 资源维护前，平台必须计算受影响的模型实例和应用SLO。
6. 租户欠费或预算耗尽时，应按策略执行告警、限速、暂停或保底服务，而非直接粗暴断流。

### 7.2 关联数据模型

```text
Tenant
 └─ Project
     ├─ Application
     │   ├─ Credential
     │   ├─ Quota/Budget
     │   └─ Route → ModelService
     └─ ModelService
         └─ ModelVersion
             └─ Deployment
                 └─ Replica
                     └─ Allocation
                         └─ Accelerator/Node/ResourcePool
```

横向关联：UsageRecord、CostRecord、PriceRule、BillItem、Metric、Log、Trace、Alert、Incident、Audit。

## 8. 配额体系

### 8.1 配额层级

```text
平台总容量
→ 租户配额
→ 项目配额
→ 应用/模型服务配额
→ API Key或用户限额
```

### 8.2 配额类型

| 类型 | 示例 | 主要用途 |
|---|---|---|
| 硬资源配额 | GPU卡数、显存、CPU、内存、存储 | 限制可占用容量 |
| 调度配额 | 队列并发、作业数、预约时长 | 控制任务进入资源池 |
| 推理配额 | 并发请求、RPM、TPM、上下文、输出Token | 保护推理服务 |
| 缓存配额 | KV Cache、Prefix Cache | 防止长会话挤占显存 |
| 商业额度 | 预算、预付余额、套餐Token | 控制消费金额 |
| 权限额度 | 可用模型、区域、网络、数据等级 | 控制访问范围 |

### 8.3 配额状态

必须区分：总额度、已分配、已使用、已预留、可借用、被冻结、超额、即将到期。

### 8.4 配额动作与策略

- 申请、审批、调整、回收和到期。
- 父子级额度不得重复计算。
- 支持临时借用，但必须有归还时间和抢占规则。
- 超限动作支持告警、排队、限速、拒绝和降级。
- 所有配额变更必须记录操作者、原因、生效时间和影响范围。

## 9. 用量、成本、价格、计费和账单闭环

### 9.1 五个概念必须分开

- 用量：实际消耗了多少资源或服务。
- 成本：平台为了提供服务付出了多少。
- 价格：对客户或内部成本中心采用什么收费规则。
- 费用：用量应用价格规则后的金额。
- 账单：在结算周期内冻结、调整并确认后的应付记录。

### 9.2 用量采集

采集维度包括：

- 卡时、实例时长、预留时长。
- 输入、输出、缓存Token和请求次数。
- 存储容量时、网络流量和日志容量。
- 模型、版本、应用、Key、租户、项目、区域和资源规格。

要求：可追溯、可幂等、可补采、可校验、可重算。

### 9.3 成本归因

```text
硬件折旧 + 电力制冷 + 机房与网络 + 软件许可 + 运维人力
→ 资源池成本
→ GPU/实例单位成本
→ 模型服务成本
→ 应用/租户成本
```

共享成本可按卡时、显存时长、Token、占用比例或约定权重分摊，并明确规则版本。

### 9.4 价格模型

- 包年包月。
- 按量阶梯价。
- 预付费额度包。
- 专属实例保底加超量。
- 免费额度与试用。
- SLA溢价。
- 内部成本中心分摊。

### 9.5 账单状态机

```text
实时预估
→ 待出账
→ 账单冻结
→ 待确认
→ 调整中
→ 已确认
→ 已结算
```

异常分支：争议、补扣、退款、冲正、坏账、重开。

### 9.6 欠费与预算控制

建议分级处理：

1. 达到70%：提醒。
2. 达到90%：强提醒并建议扩容或降级。
3. 达到100%：根据业务等级执行限速、禁止新任务或保留关键服务。
4. 超过宽限期：暂停非关键调用，并保留审计和数据导出能力。

## 10. 运营指标树

### 10.1 北极星目标

```text
在满足质量、安全和SLO的前提下，提高有效AI服务产出和单位算力毛利。
```

### 10.2 指标分层

| 层级 | 核心指标 |
|---|---|
| 用户体验 | 模型上线时长、API开通时长、首次调用成功率、TTFT、P95/P99延迟 |
| 模型质量 | 任务成功率、质量评分、幻觉/安全命中、版本回滚率 |
| 资源效率 | GPU有效利用率、显存利用率、碎片率、排队时长、缓存命中率 |
| 稳定性 | API可用率、OOM率、扩缩容失败率、MTTR、错误预算 |
| 成本经营 | 每百万Token成本、每卡收入、空闲成本、毛利率、预算消耗率 |
| 客户运营 | 活跃应用、调用留存、套餐续费、扩容率、流失率 |
| 产品效率 | 关键功能使用率、审批自动化率、告警处置闭环率 |

### 10.3 指标不能孤立使用

- GPU利用率高不等于用户体验好。
- QPS不能跨不同上下文长度直接比较。
- Token量增长不等于收入和毛利同步增长。
- 平均延迟不能代表P99长尾。
- 功能上线不等于用户旅程完成。

## 12. PRD验收要点

1. 任意资源、模型和应用均有明确状态、允许动作和异常分支。
2. 页面不可只显示失败，必须提供失败原因、影响对象和建议动作。
3. 模型下线、资源维护和应用注销均必须完成依赖检查。
4. 配额扣减、借用、回收和超限动作必须可审计。
5. 用量明细能够追溯到租户、项目、应用、模型、版本、Key和资源规格。
6. 账单支持冻结、调整、冲正和重算，不直接修改历史原始用量。
7. 每个核心旅程都有完成率、时长、失败率和满意度指标。

## 13. 面试讲法

```text
我不会把MaaS理解为模型上传和一键部署，而是把它看成三条生命周期的交汇：
资源生命周期决定算力是否可用，模型生命周期决定服务是否可信，应用生命周期决定能力是否真正被业务消费。
平台通过租户、项目、应用、模型服务、部署和资源分配的数据模型将三条链路贯通，再通过配额、用量、成本、价格、账单和运营指标形成经营闭环。
这样做的价值是，每次故障、成本异常或扩容决策都能追溯到具体应用、模型版本和资源，而不是停留在孤立的GPU或Pod指标。
```
