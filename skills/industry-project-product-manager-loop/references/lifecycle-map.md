# 行业项目产品经理生命周期与文档地图

## 1. 总体链路

```text
行业与机会
  ↓
商业与市场
  ↓
客户与用户
  ↓
竞争与定位
  ↓
立项与规划
  ↓
方案与需求
  ↓
工程与集成
  ↓
测试与上线
  ↓
运营与复盘
```

文档不是越多越好。每份文档必须对应一个决策、审批、协作或交付目的。

## 2. 阶段与核心问题

| 阶段 | 核心问题 | 典型交付物 | 主要读者 |
|---|---|---|---|
| 0 项目进入 | 这是什么项目，需要做到什么深度？ | Brief、状态文件、范围草案、证据清单 | 项目发起人、产品 |
| 1 行业机会 | 为什么现在做，问题是否真实？ | 行业研究、价值链、痛点、机会分析 | 产品、战略、售前 |
| 2 商业市场 | 值不值得投入，客户是否会买？ | BP、BRD、MRD、商业模式、ROI | 管理层、业务、销售 |
| 3 用户研究 | 谁购买、谁使用、要完成什么任务？ | 调研报告、Persona、JTBD、旅程 | 产品、设计、研发 |
| 4 竞争策略 | 替代方案是什么，我们为何胜出？ | 竞品深评、定位、Build/Buy/Integrate | 产品、战略、架构 |
| 5 立项规划 | 做什么、不做什么、谁负责、何时交付？ | PID、范围、RACI、Roadmap、需求池 | 项目组、管理层 |
| 6 产品方案 | 产品具体如何解决问题？ | 总体方案、架构、流程、PRD、原型 | 研发、测试、客户 |
| 7 工程集成 | 系统如何实现并与现有环境协同？ | SRS/SRD、HLD、SID、安全、部署 | 架构、研发、运维 |
| 8 测试上线 | 如何证明可用并安全发布？ | 测试、UAT、上线、回滚、培训 | 测试、业务、运维 |
| 9 运营复盘 | 是否有效，下一轮做什么？ | 数据报告、复盘、路线图更新 | 产品、管理层、客户成功 |

## 3. 文档选择规则

### BP

在需要证明“这是一门可以成立的生意”时使用。

必须回答：客户、价值、市场、竞争、收入、成本、获客、资源、风险和里程碑。

### BRD

在需要公司批准投入或业务方确认项目价值时使用。

必须回答：现状、业务问题、目标、范围、价值、指标、投入、风险和立项建议。

### MRD

在需要选择市场、客户和定位时使用。

必须回答：市场结构、细分客户、需求强度、竞争格局、产品定位、进入策略和优先级。

### PID / Project Charter

在项目正式启动，需要明确治理方式时使用。

必须回答：目标、范围、交付物、组织、里程碑、预算、风险、依赖和验收。

### PRD

在需要研发、测试、设计和业务对“产品做什么”形成一致理解时使用。

必须包含：用户、场景、流程、功能、规则、状态、异常、权限、数据、非功能、埋点和验收。

### SRS / SRD

在正式软件交付、招投标、外包或大型系统项目中使用。

强调系统边界、接口、性能、安全、兼容、部署和工程约束。

### SID

在存在多个外部系统、设备、数据源或协议对接时使用。

必须包含：系统边界、接口清单、字段映射、鉴权、时序、错误码、重试、幂等、安全和联调。

## 4. 交付物依赖

```text
行业证据
  ├─→ 痛点分析
  │     ├─→ BRD
  │     └─→ 用户研究计划
  ├─→ MRD
  └─→ 竞品选择

BRD + MRD + 用户研究 + 竞品
  └─→ 产品定位与PID

PID + 用户旅程 + 需求池
  └─→ 总体方案与PRD

PRD + 架构约束 + 对接系统
  └─→ SRS/SRD/SID

PRD验收标准 + SRS非功能要求
  └─→ 测试/UAT/上线方案

上线数据 + 用户反馈 + 故障记录
  └─→ 复盘与下一版Roadmap
```

## 5. 角色责任建议

| 交付物 | 产品 | 业务/销售 | 设计 | 架构/研发 | 测试 | 运维/安全 |
|---|---:|---:|---:|---:|---:|---:|
| 行业机会 | A/R | C | C | C | I | C |
| BP/BRD/MRD | A/R | R/C | I | C | I | C |
| 用户研究 | A/R | C | R/C | I | I | I |
| 竞品分析 | A/R | C | C | C | I | I |
| PID/Roadmap | A/R | C | I | C | C | C |
| PRD | A/R | C | R | C | C | C |
| SRS/SID | C | I | I | A/R | C | C |
| 测试/UAT | A/C | C | I | C | A/R | C |
| 上线方案 | C | I | I | R | C | A/R |
| 复盘 | A/R | C | C | C | C | C |

A：最终负责，R：执行，C：协作评审，I：知会。

## 6. 完成度判断

每个阶段按四个层级评估：

- `0 缺失`：没有相关材料；
- `1 草稿`：有结构但关键结论或证据不足；
- `2 可评审`：核心内容完整，可进入跨角色评审；
- `3 已确认`：结论获得负责人或客户确认；
- `4 已验证`：上线或项目数据证明结论有效。

项目早期不要求所有文档达到4，但关键上游结论必须至少达到2，才能支撑下游设计。

## 7. 常见错误

- 先写完整PRD，再寻找用户痛点；
- 把目标用户只写成“企业客户”或“运维人员”；
- 竞品分析只有功能打勾，没有产品决策；
- 把产品目标写成已经实现的收益；
- Roadmap只是功能排期，没有阶段目标；
- 只有主流程，没有异常、权限和状态；
- 测试用例和PRD验收标准无法对应；
- 上线方案没有监控与回滚；
- 项目结束后没有结果数据和复盘。
