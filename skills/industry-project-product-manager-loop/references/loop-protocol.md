# 产品经理 Loop Engineering Protocol

## 1. 目标

让Agent不依赖用户逐步提醒，而是在明确边界内持续完成项目级产品工作：发现缺口、选择行动、生成材料、校验结果、记录状态，并判断是否继续。

## 2. Loop对象

每个Loop围绕一个项目状态对象运行：

- 目标：本项目最终需要达到什么完成档位；
- 当前状态：处于哪个生命周期阶段；
- 工作队列：待完成、待验证和待决策事项；
- 证据库：事实、访谈、数据、外部来源；
- 交付物：已有文件和质量状态；
- 决策日志：关键取舍及原因；
- 风险：影响范围、概率、负责人和应对措施；
- 退出条件：何时停止。

## 3. 单轮协议

### Step 1：读取状态

读取项目目录、项目状态、最近变更和未完成事项。

### Step 2：扫描变化

判断本轮新增了什么：

- 新客户信息；
- 新数据；
- 新竞品信息；
- 新需求；
- 技术约束变化；
- 范围变化；
- 质量门失败；
- 上线结果。

### Step 3：建立候选行动

每项行动记录：

- 解决的问题；
- 所属阶段；
- 预期产物；
- 所需证据；
- 依赖；
- 风险；
- 是否需要人工决策。

### Step 4：排序

默认优先级：

1. 阻塞项目的关键决策；
2. P0需求或上线安全问题；
3. 高风险且证据不足的假设；
4. 上下游文档冲突；
5. 必选交付物缺失；
6. 可提升完整度的增强项。

### Step 5：执行

每轮聚焦一个主目标或少量强相关目标。避免一次创建大量彼此缺少证据的文档。

### Step 6：验证

检查：

- 产物是否回答了对应决策问题；
- 是否有证据；
- 是否与已有材料一致；
- 是否产生了新的风险或依赖；
- 是否可由后续角色使用；
- 是否满足质量门。

### Step 7：更新状态

更新：

- 当前阶段；
- 已完成项；
- 失败项；
- 新假设；
- 新风险；
- 下一步；
- 人工决策；
- 完成度。

### Step 8：判断

输出三种状态之一：

- `continue`：有可执行的高价值动作；
- `human_required`：需要明确的业务、预算、法律、架构或客户决策；
- `complete`：所选档位全部通过；
- `evidence_exhausted`：缺少外部证据，继续生成只会重复假设。

## 4. 外循环与内循环

### 外循环：项目生命周期

从行业研究持续推进到复盘，决定当前最应该推进哪个阶段。

### 内循环：单项交付物

例如PRD内循环：

```text
读取上游需求
→ 生成用户故事和流程
→ 检查规则、状态、异常、权限、数据
→ 生成验收标准
→ 执行可测试性检查
→ 修正
→ 通过后退出
```

竞品分析内循环：

```text
确定竞争问题
→ 选择竞品
→ 获取可靠信息
→ 统一维度比较
→ 识别边界和替代关系
→ 形成Build/Buy/Integrate决策
→ 检查是否能指导产品取舍
```

## 5. 人在回路

以下事项必须升级人工：

- 项目是否立项；
- 预算与投入承诺；
- 商业价格；
- 法律合规解释；
- 生产高风险操作；
- 核心架构取舍；
- 客户合同承诺；
- 相互冲突但没有决策人的优先级；
- 公开发布未经确认的数据。

Agent应提供选项、影响、证据和推荐，不得伪造授权。

## 6. 防失控机制

### 最大循环次数

对于同一交付物，连续三轮没有新增证据或实质改进时停止，转为证据获取计划。

### 防重复

新建文件前检查是否已有同类内容。优先更新、合并或建立索引。

### 防文档膨胀

只有满足以下任一条件才独立成文档：

- 面向不同决策人；
- 生命周期阶段不同；
- 需要单独审批；
- 内容复杂且会频繁维护；
- 对外交付格式有要求。

### 防伪完成

文件存在不等于完成。完成必须同时满足：

- 内容非空且结构完整；
- 关键结论有证据或明确标记假设；
- 与上游和下游可追溯；
- 有负责人或使用对象；
- 有验收或下一步。

## 7. 循环评分

每轮可以按以下维度评分，每项0—5分：

- 证据增量；
- 决策增量；
- 风险降低；
- 交付物质量提升；
- 下游可执行性。

总分低于5且没有新证据时，不应机械继续。

## 8. 项目结束后的回灌

完成项目后记录：

- 哪些输入最容易缺失；
- 哪些文档实际无人使用；
- 哪些检查发现了高价值问题；
- 哪些模板应调整；
- 哪些行业知识可复用；
- 哪些自动化脚本需要增强；
- 哪些测试场景应加入Skill。
